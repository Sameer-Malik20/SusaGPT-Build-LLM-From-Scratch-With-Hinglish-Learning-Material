# Database Scaling: Complete Guide for Fullstack Engineers

## Hinglish Explanation (Beginner Level)

Bhai, database scaling matlab tumhari database ko handle karne ki capacity badhana jab tumhare users aur data grow hote hain.

Socho tumne ek chhoti si website banayi with 100 users - ek simple database kaafi hai. Lekin jab 1 lakh users aayein, toh database slow ho jayega. Tab tumhe scaling karni padegi!

**Two types ki scaling hoti hai:**

1. **Vertical Scaling (Scale Up)** - Server ko powerful banana
   - More RAM, CPU, Storage add karna
   - Simple but limited
   - AWS: t3.medium se t3.xlarge mein upgrade

2. **Horizontal Scaling (Scale Out)** - Multiple servers add karna
   - Data ko multiple machines mein distribute karna
   - Complex but unlimited potential
   - Example: 3 server, har ek 1/3 data handle kare

**Scaling strategies:**
- **Read Replicas** - Slow reads ko distribute karo
- **Sharding** - Data ko chunk mein baato
- **Caching** - Redis jaise cache use karo
- **Connection Pooling** - Connections efficiently manage karo

Real-world mein companies jaise YouTube, Facebook, Google sabhi scaling techniques use karti hain!

## Deep Technical Explanation

Database scaling encompasses strategies and techniques to increase database capacity, throughput, and performance as data volume and user load grow.

### PostgreSQL Replication with Prisma

```typescript
// src/config/database.ts
import { PrismaClient } from '@prisma/client';

// Primary database (writes)
const primaryDb = new PrismaClient({
  datasources: {
    db: {
      url: process.env.DATABASE_PRIMARY_URL,
    },
  },
  log: ['error'],
});

// Replica database (reads)
const replicaDb = new PrismaClient({
  datasources: {
    db: {
      url: process.env.DATABASE_REPLICA_URL,
    },
  },
  log: ['error'],
});

// Read-Write Splitting Service
export class DatabaseService {
  // All writes go to primary
  async create<T extends keyof typeof primaryDb>(model: T, data: any) {
    return primaryDb[model].create({ data });
  }

  async update<T extends keyof typeof primaryDb>(
    model: T,
    where: any,
    data: any
  ) {
    return primaryDb[model].update({ where, data });
  }

  async delete<T extends keyof typeof primaryDb>(model: T, where: any) {
    return primaryDb[model].delete({ where });
  }

  // All reads go to replica
  async findMany<T extends keyof typeof replicaDb>(
    model: T,
    options?: any
  ) {
    return replicaDb[model].findMany(options);
  }

  async findUnique<T extends keyof typeof replicaDb>(
    model: T,
    options?: any
  ) {
    return replicaDb[model].findUnique(options);
  }

  async count<T extends keyof typeof replicaDb>(
    model: T,
    options?: any
  ) {
    return replicaDb[model].count(options);
  }

  // For read-after-write consistency
  async findById<T extends keyof typeof primaryDb>(
    model: T,
    id: string
  ) {
    return primaryDb[model].findUnique({ where: { id } });
  }
}

export const db = new DatabaseService();

// Usage
async function createUser(data: CreateUserInput) {
  // Write to primary
  const user = await db.create('user', data);
  return user;
}

async function getUser(id: string) {
  // Read from replica
  return db.findUnique('user', { where: { id } });
}

async function getUsers(page: number) {
  // Read from replica
  return db.findMany('user', {
    skip: (page - 1) * 20,
    take: 20,
    orderBy: { createdAt: 'desc' },
  });
}
```

```typescript
// src/config/mongodb-sharded.ts
import mongoose from 'mongoose';

// Sharded cluster connection
const MONGODB_SHARDED_URI = process.env.MONGODB_SHARDED_URI ||

export async function connectToShardedCluster() {
  await mongoose.connect(MONGODB_SHARDED_URI, {
    // MongoDB driver options
    maxPoolSize: 100,
    minPoolSize: 10,
    maxIdleTimeMS: 30000,
    serverSelectionTimeoutMS: 30000,
    socketTimeoutMS: 45000,
    
    // Read preference
    readPreference: 'secondaryPreferred',
    
    // Write concern
    w: 'majority',
    wtimeout: 5000,
  });

// User model with user_id as shard key
const userSchema = new mongoose.Schema({
  _id: mongoose.Schema.Types.ObjectId,
  email: { type: String, required: true },
  name: { type: String, required: true },
  createdAt: { type: Date, default: Date.now },
}, {
  collection: 'users', // Collection is sharded on user_id
});

// Compound shard key for better distribution
const orderSchema = new mongoose.Schema({
  _id: mongoose.Schema.Types.ObjectId,
  userId: { type: mongoose.Schema.Types.ObjectId, required: true },
  total: Number,
  status: String,
  createdAt: { type: Date, default: Date.now },
}, {
  collection: 'orders',
  // Compound shard key: userId + createdAt for range queries per user
```

### Redis Cluster for Distributed Caching

```typescript
// src/config/redis-cluster.ts
import Redis from 'ioredis';

// Redis Cluster configuration
const redisClusterNodes = [
  { host: 'redis-1.example.com', port: 6379 },
  { host: 'redis-2.example.com', port: 6379 },
  { host: 'redis-3.example.com', port: 6379 },
  { host: 'redis-4.example.com', port: 6379 },
  { host: 'redis-5.example.com', port: 6379 },
  { host: 'redis-6.example.com', port: 6379 },
];

export const redisCluster = new Redis.Cluster(redisClusterNodes, {
  // Cluster options
  clusterRetryStrategy: (times: number) => {
    const delay = Math.min(times * 100, 3000);
    return delay;
  },
  
  // Scale reads to replicas
  readOnly: true,
  enableReadyCheck: true,
  
  // Retry configuration
  maxRetriesPerRequest: 3,
  retryDelayOnFailover: 100,
  
  // Redistribute slots automatically
  lazyConnect: false,
});

// Pipeline for batch operations
async function batchGet(keys: string[]) {
  const pipeline = redisCluster.pipeline();
  
  for (const key of keys) {
    pipeline.get(`cache:${key}`);
  }
  
  const results = await pipeline.exec();
  return results?.map(([err, value]) => value);
}

// Consistent hashing for custom sharding
import { ConsistentHash } from './consistent-hash';

const ring = new ConsistentHash(redisClusterNodes);

function getNode(key: string) {
  return ring.getNode(key); // Returns { host, port }
}

async function setWithNode(key: string, value: any, ttl?: number) {
  const node = getNode(key);
  const nodeClient = new Redis(`${node.host}:${node.port}`);
  
  await nodeClient.setex(`cache:${key}`, ttl || 3600, JSON.stringify(value));
  await nodeClient.quit();
}
```

```sql
-- Create publication on primary (US-East)
CREATE PUBLICATION app_publication FOR ALL TABLES WITH (publish = 'insert, update, delete');

-- Create subscription on replica (EU-West)
CREATE SUBSCRIPTION eu_replica

WITH (copy_data = true, slot_name = 'eu_replica_slot');

ALTER SUBSCRIPTION eu_replica SET (conflict_log = true);

SELECT 
    application_name,
    state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    (sent_lsn - replay_lsn) AS replication_lag
```

```typescript
  { id: 1, range: [10000, 19999], connection: 'postgres://shard1.db.com:5432' },
  { id: 2, range: [20000, 29999], connection: 'postgres://shard2.db.com:5432' },
  { id: 3, range: [30000, 39999], connection: 'postgres://shard3.db.com:5432' },
];

// Determine shard for a user ID
function getShard(userId: number) {
  for (const shard of shards) {
    if (userId >= shard.range[0] && userId <= shard.range[1]) {
      return shard;
    }
  }
  return shards[0]; // Default
}

// Sharded database service
class ShardedDatabaseService {
  private connections: Map<number, PrismaClient> = new Map();
  
  async getConnection(shardId: number): Promise<PrismaClient> {
    if (this.connections.has(shardId)) {
      return this.connections.get(shardId)!;
    }
    
    const shard = shards.find(s => s.id === shardId);
    const client = new PrismaClient({
      datasources: { db: { url: shard!.connection } },
    });
    
    this.connections.set(shardId, client);
    return client;
  }
  
  // Route to correct shard
  async getUser(userId: number) {
    const shard = getShard(userId);
    const db = await this.getConnection(shard.id);
    return db.user.findUnique({ where: { id: userId } });
  }
  
  // Cross-shard query (expensive - use sparingly)
  async getAllUsersCount() {
    const counts = await Promise.all(
      shards.map(s => this.getConnection(s.id).then(db => 
        db.user.count()
      ))
    );
    return counts.reduce((sum, count) => sum + count, 0);
  }
  
  // Fan-out write
  async createUserOnAllShards(userData: any, userId: number) {
    const shard = getShard(userId);
    const db = await this.getConnection(shard.id);
    return db.user.create({ data: { id: userId, ...userData } });
  }
}

// Alternative: Hash-based sharding
function getShardByHash(userId: string) {
  const hash = crypto.createHash('md5').update(userId).digest('hex');
  const hashNum = parseInt(hash.substring(0, 8), 16);
  return shards[hashNum % shards.length];
}
```

### Example 3: Caching Strategy with Redis

```typescript
// Multi-level caching strategy

class CachingService {
  private redis: typeof redis;
  private localCache: Map<string, { value: any; expiry: number }> = new Map();
  private localCacheTTL = 1000; // 1 second local cache

  constructor(redisClient: typeof redis) {
    this.redis = redisClient;
    
    // Cleanup local cache periodically
    setInterval(() => this.cleanupLocalCache(), 1000);
  }

  async get<T>(key: string): Promise<T | null> {
    // 1. Check local cache (fastest)
    const local = this.localCache.get(key);
    if (local && local.expiry > Date.now()) {
      return local.value;
    }

    // 2. Check Redis
    const redisValue = await this.redis.get(`cache:${key}`);
    if (redisValue) {
      const value = JSON.parse(redisValue);
      // Populate local cache
      this.localCache.set(key, { value, expiry: Date.now() + this.localCacheTTL });
      return value;
    }

    return null;
  }

  async set(key: string, value: any, ttl: number = 3600): Promise<void> {
    // Set in Redis
    await this.redis.setex(`cache:${key}`, ttl, JSON.stringify(value));
    // Set in local cache
    this.localCache.set(key, { value, expiry: Date.now() + this.localCacheTTL });
  }

  async invalidate(pattern: string): Promise<void> {
    // Clear local cache
    for (const key of this.localCache.keys()) {
      if (key.includes(pattern)) {
        this.localCache.delete(key);
      }
    }
    
    // Clear Redis
    const keys = await this.redis.keys(`cache:${pattern}`);
    if (keys.length > 0) {
      await this.redis.del(...keys);
    }
  }

  private cleanupLocalCache(): void {
    const now = Date.now();
    for (const [key, { expiry }] of this.localCache.entries()) {
      if (expiry < now) {
        this.localCache.delete(key);
      }
    }
  }
}

// Cache patterns
async function getUserWithCache(userId: string) {
  const cacheKey = `user:${userId}`;
  
  // Try cache
  const cached = await cachingService.get<User>(cacheKey);
  if (cached) return cached;
  
  // Fetch from database
  const user = await db.findUnique('user', { where: { id: userId } });
  
  // Cache for 5 minutes
  await cachingService.set(cacheKey, user, 300);
  
  return user;
}

// Write-through cache
async function updateUserWithCache(userId: string, data: any) {
  // Update database
  const user = await db.update('user', { where: { id: userId }, data });
  
  // Update cache
  await cachingService.set(`user:${userId}`, user, 300);
  
  return user;
}
```

## Tradeoffs

### SQL vs NoSQL Scaling

| Aspect | SQL (PostgreSQL) | NoSQL (MongoDB) |
|--------|-----------------|-----------------|
| Scaling Model | Vertical + Read Replicas | Horizontal Sharding |
| Sharding Complexity | Application-level | Built-in |
| Joins Across Shards | Difficult | Limited |
| ACID | Full support | Limited (per-shard) |
| Consistency | Strong | Eventual (default) |
| Best For | Complex queries, OLTP | Write-heavy, flexible schema |

### Replication vs Sharding

| Aspect | Replication | Sharding |
|--------|------------|----------|
| Purpose | High availability, read scale | Write scale |
| Data | Full copy on each node | Partitioned |
| Writes | Single primary | Distributed |
| Complexity | Lower | Higher |
| Consistency | Strong | May need tradeoffs |

## Interview Questions

### Q1: How would you scale a database from 1GB to 1TB?

**Answer**: Scale vertically first with better hardware, add read replicas for read-heavy workloads, implement caching with Redis, then shard when writes become bottleneck. Consider partitioning for time-series data, use read replicas for analytics.

### Q2: What is the CAP theorem?

**Answer**: In distributed systems, you can only guarantee 2 of 3: Consistency (all nodes see same data), Availability (every request gets response), Partition tolerance (system works despite network failures). SQL typically prioritizes C, NoSQL often prioritizes A/P.

### Q3: What is eventual consistency?

**Answer**: Model where updates propagate asynchronously to replicas. Clients may see stale data temporarily but eventually all copies converge. Tradeoff for higher availability and performance. Use read-after-write or synchronous replication when strong consistency needed.

### Q4: How does sharding differ from partitioning?

**Answer**: Partitioning divides large tables within single database (sub-tables). Sharding distributes data across separate database instances/servers. Partitioning is transparent to application, sharding requires application-level routing.

### Q5: What is read-after-write consistency?

**Answer**: Guarantee that after a write, subsequent reads return the updated value. Achieved by reading from primary after writes, or waiting for replication to catch up. Critical for operations like posting then viewing.

### Q6: How would you handle a primary failure?

**Answer**: With automatic failover using tools like Patroni, Consul, or cloud-managed failover. Replica detects primary failure, promotes itself, updates routing. Applications reconnect to new primary. Client-side retries handle brief unavailability.

### Q7: What are the tradeoffs of sharding?

**Answer**: Complexity increases significantly, cross-shard queries are expensive, resharding is challenging, joins across shards limited, operational complexity increases. Use only when necessary and plan shard keys carefully.

### Pattern 4: Multi-Cloud Database

```typescript
// Multi-cloud deployment for disaster recovery
class MultiCloudDatabase {
  private databases = {
    aws: new PrismaClient({ datasourceUrl: process.env.AWS_DB_URL }),
    gcp: new PrismaClient({ datasourceUrl: process.env.GCP_DB_URL }),
    azure: new PrismaClient({ datasourceUrl: process.env.AZURE_DB_URL }),

    const results = await Promise.allSettled([
      this.databases.aws.create({ data }),
      this.databases.gcp.create({ data }),
      this.databases.azure.create({ data }),
    ]);
    
    // Check for failures
    const failures = results.filter(r => r.status === 'rejected');
    if (failures.length > 1) {
      throw new Error('Multi-cloud write failed');
    }
    
    return results[0];
  }
  
  async read(cloud: 'aws' | 'gcp' | 'azure' = 'aws') {
    return this.databases[cloud].findMany();
  }
```
